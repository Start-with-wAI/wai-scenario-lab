# Repository Scaffold

This document explains the current repository structure for the wAI Scenario Lab capstone build. It is intended for Verónica, Jason, future contributors, and Kaggle reviewers who need to understand where the active implementation lives and how the project is organized.

## Source of Truth

The active runnable implementation is under:

```text
core-lab/
```

The repository may still contain planning notes or historical branch artifacts from earlier Day 1 work, but the capstone application should be built, tested, and documented from `core-lab/` unless the team explicitly changes that decision.

## Top-Level Repository

```text
wai-scenario-lab/
├── .agents/                  # Agent skill notes and future portable skill assets
├── core-lab/                 # Active runnable application and capstone source of truth
├── README.md                 # Repository-level overview and judge-facing entry point
```

The top-level README should direct reviewers into `core-lab/` for the runnable project.

## Application Code

```text
core-lab/app/
├── agent.py                  # Root ADK/agents-cli app entry point
├── workflow.py               # Four-agent workflow and deterministic routing
├── schemas.py                # Pydantic data contracts for agent outputs and Scenario Briefs
├── fast_api_app.py           # FastAPI server wrapper generated by the scaffold
└── app_utils/                # Scaffold utilities for services, telemetry, A2A, and adapters
```

### Key responsibilities

- `agent.py` should represent the wAI Scenario Lab, not a generic sample assistant.
- `workflow.py` should preserve the four-agent sequence: Scenario Guide, Workflow Analysis, Value and Evidence, and Safety and Quality Review.
- `schemas.py` is the contract layer that keeps outputs structured, reviewable, and testable.
- `wai_scenario_config.json` is the scenario content source of truth. Scenario questions, guardrails, episode CTAs, measurements, and disclosures should come from configuration rather than being scattered through application code.

## MCP Scaffold

```text
core-lab/mcp_server/
└── roi_calculator_server.py  # Existing MCP server to be refactored in Phase 4
```

The current MCP server should be treated cautiously because ROI, opportunity cost, dollar savings, and marketing-equity calculations can exceed the MVP's responsible-use boundary. Phase 4 should refactor or replace this with a scenario configuration and safe measurement MCP server.

Preferred future MCP role:

```text
core-lab/mcp_server/scenario_config_server.py
```

Expected safe responsibilities:

- List available scenarios.
- Return scenario metadata.
- Return configured questions.
- Return configured measurement definitions.
- Select a primary or fallback observation measure from validated inputs.

It should not calculate ROI, dollar savings, opportunity cost, annual value, five-year value, marketing equity, or guaranteed productivity gains.

## Test Scaffold

```text
core-lab/tests/
├── eval/                     # Evaluation datasets and config
├── integration/              # End-to-end or service-level tests
├── scenarios.feature         # Gherkin scenario coverage
└── unit/                     # Deterministic unit tests
```

Expected test coverage should include:

- Configuration loading.
- Stable scenario IDs.
- Four questions per MVP scenario.
- Numeric validation.
- Sensitive-data handling.
- High-risk request handling.
- One-action restraint.
- One-measurement restraint.
- Required human-review language.
- Required responsible-use limitation.
- Scenario-specific podcast CTA matching.

The target evaluation set is 18 cases, with six cases per scenario.

## Scenario Configuration

The active configuration file is:

```text
core-lab/wai_scenario_config.json
```

It should contain only the MVP scenarios:

1. `cool_down_tax`
2. `brain_fog`
3. `blank_page`

Each scenario should define:

- Scenario metadata
- Episode metadata
- Four guided questions
- Validation messages
- Allowed next-action categories
- Guardrails
- Primary and fallback measurements
- Episode CTA

## Branch Reconciliation Rule

The `veronica` branch contains valuable Day 1 planning, schema, workflow, and documentation work. However, it used a root-level scaffold shape.

The integration rule is:

```text
Do not directly merge duplicate root-level app, docs, tests, or mcp_server structures into main.
Port useful assets into core-lab instead.
```

This keeps the repository understandable and prevents Kaggle reviewers from seeing two competing implementations.

## Capstone Reviewer Notes

The scaffold should make three things clear:

1. The project is a configuration-driven multi-agent prototype.
2. The app intentionally limits output to one next action and one measurement.
3. Safety and commercial-boundary controls are part of the architecture, not afterthoughts.

The final repository should remain small, reproducible, and free of secrets or private client data.
